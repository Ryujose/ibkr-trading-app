- [x] **Task #73** — Documentation (Task H) — This file. The main GLFW window is borderless (`GLFW_DECORATED=GLFW_FALSE`) so the app implements its own edge-resize via `glfwGetCursorPos(g_AppWindow, ...)` + edge-zone math. With `ImGuiConfigFlags_ViewportsEnable` on, an undocked window becomes a separate OS-level GLFW window that can sit over (or past) the main window's borders. `glfwGetCursorPos` still returns coordinates relative to the main window in that case, so `mx > ww - kEdge` / `mx < kEdge` / `my > wh - kEdge` would all evaluate true when the cursor is on a floating viewport at the edge — and `glfwSetWindowPos/Size(g_AppWindow, ...)` would resize the main window every frame the user dragged the floating window. Fix: gate the edge-detection block behind `glfwGetWindowAttrib(g_AppWindow, GLFW_HOVERED)` (false when another window is between the cursor and the main window's content), with an `s_resizing` exemption so a resize started on a real main-window edge can drag past the edge without snapping back. The static drag-state declarations (`s_resizing`, `s_edge`, `s_startMX/MY/WX/WY/WW/WH`) were hoisted to the top of the function so the early-return can consult `s_resizing` before any of the other math runs; the cursor is reset to `nullptr` on the early-return path so a stale resize cursor doesn't linger when the cursor re-enters the main window.
